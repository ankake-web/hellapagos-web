# ヘルパゴス Web版 — 設計書

> ニセ協力サバイバルゲーム「ヘルパゴス（Hellapagos / Gigamic, 2017）」のオンライン対戦Web版の設計。
> 本書は「ゲームルールの整理」「MVPスコープ」「アーキテクチャ」「データモデル」「通信仕様」「AI設計」「画面設計」「実装ロードマップ」をまとめる。

---

## 1. ゲーム概要

- **ジャンル**: ニセ協力（co-opetition）／正体隠匿寄りのサバイバル・パーティゲーム
- **人数**: 3〜12人（多人数ほど盛り上がる）
- **プレイ時間**: 約20分
- **テーマ**: 無人島に漂着した生存者たちが、ハリケーン到来までに**いかだ**を造って脱出する。だが水・食料が足りなければ「誰かを生贄にして減らす」しかない。
- **本質**: 表向きは全員協力。実際は資源不足の瞬間に**投票による追放（生贄）**と**隠し財産（個人で水・食料を隠す）**が牙を剥く。

### 勝敗
- **勝利**: ハリケーン到来までに、生存人数分の「いかだの席」＋航海用の「水・食料」を確保して脱出する。
- **敗北**: 嵐に飲まれる／全員が渇き・飢えで死ぬ／いかだに乗れない。
- **ソロサバイバル・バリアント**: 自分だけが脱出すれば勝ち（最も殺伐とする）。

## 2. 完全版ルール（調査結果の整理）

1ラウンド＝1日。以下のサイクルを繰り返す。

### 2.1 天候フェイズ
- 天候カードを1枚めくる。雨は水確保に有利。
- **嵐（ハリケーン）**が出ると、そのラウンド終了時までに脱出できなければ全員敗北（カウントダウンの終端）。

### 2.2 行動フェイズ
各プレイヤーは1日1回、4つの行動から1つを選ぶ。

| 行動 | 効果 |
|------|------|
| **魚釣り** | 袋からボールを1個引く。描かれた魚の数（1〜3）だけ食料トラックを進める |
| **水汲み** | 天候に応じた量の水を確保 |
| **木材集め** | いかだ用の木材を自動で1進める。さらにボールを引いて追加可能だが、黒ボール（ヘビ）を引くと毒で病気になり次ターン行動不能 |
| **難破船の探索** | 山札から「漂着物カード」を1枚引いて手札（＝個人の隠し財産）に加える |

### 2.3 カードプレイ（任意・割り込み可）
漂着物カードは自他のターンに割り込んで使える。3系統：
- **資源カード**: 水／食料（生存判定で供出できる。隠しておけば自分だけ助かる）
- **使い切り特殊**: 解毒剤、ブードゥー人形（死者復活）、睡眠薬（投票無効化・盗み）、フルーツバスケット（投票防御）、目覚まし時計（手番プレイヤー変更）など
- **永続**: 斧、水筒、釣り竿、水晶玉 など（行動の効率を恒常的に上げる）
- **拳銃**: 他プレイヤーを即座に射殺。鉄板（防具）でも防げないほど速い

### 2.4 生存フェイズ
- 生存者1人につき**水1・食料1**を消費する。
- 資源が不足する場合、不足数と同数のプレイヤーを**投票で追放**する。
  1. **議論**: 責任のなすり付け合い
  2. **投票**: 全員が同時に追放対象を指す
  3. **追放**: 最多票が脱落。手持ちカードは隣（または殺害者）へ渡る
  4. **同票**: その日のリーダー（トーテム保持者）が独断で決める
- 腐った魚・汚れた水などは、マッチがあるか「投票で必要と認められた配給」として消費すれば安全に食べられる。

### 2.5 脱出フェイズ
以下を**両方**満たすと脱出可能：
- いかだの席が（脱出する）生存人数分ある
- 航海用の水・食料が人数分ある（その日の食事とは別枠）

嵐が来た日にこれを満たせなければ敗北。

## 3. MVPスコープ

「行動 → 生存判定 → 投票 → 脱出」のコアループをオンライン＋AIで成立させることを最優先する。
カードの種類などは段階的に拡張する。

### MVPで実装する
- ルーム作成／URL共有での参加、ロビー、ホストによる開始
- 人数調整のための **AIボット**（行動選択・投票・脱出判断）
- 共有トラック: **食料 / 水 / 木材**、木材→いかだ席への換算
- 1日サイクル: 天候 → 同時行動 → 生存判定 → （不足時）同時投票 → 脱出判定
- 行動: 魚釣り / 水汲み / 木材集め / 難破船の探索
- 嵐カウントダウン（残り日数）。0日で最終脱出判定
- **隠し財産**: 探索で得た資源カードは個人手札。生存判定で「供出する／隠す」を選べる
- サーバ権威 ＋ 他人の手札を秘匿する**状態リダクション**
- リプレイ用シードRNG（決定論的エンジン）

### 実装済み拡張（M1 — フェイズ2の一部）
- **特殊アイテム7種**: 拳銃（射殺＋略奪）・睡眠薬（投票免疫）・解毒剤（病気回復）・ブードゥー人形（蘇生）・斧／釣り竿／浄水器（収量+1の永続）。
  探索の山札から確率で出現し、能動アイテムは `item:play` で割り込み使用。永続は所持で自動発動。
- **議論チャット**: ルーム単位のリアルタイムチャット（履歴付き）。観戦者も発言可。
- **観戦**: 進行中・満員のルームへ参加すると観戦者になり、秘匿済みの公開状態を閲覧できる。
- **再接続復帰**: localStorage のセッションで席（または観戦）へ自動復帰。切断中はAIが代行して進行を止めない。
- **演出**: 死亡・脱出・嵐などの重要イベントをポップアップ表示。

### 実装済み拡張（M2）
- **AI性格付け（BotPersona 4種）**: 協力的／溜め込み屋／狙撃手／臆病者。供出・投票・アイテム使用が変化（§9参照）。投票フェイズで性格別の煽りチャットも出す。性格は進行中は秘匿、終了時に公開。
- **ソロサバイバル・バリアント**: ロビーでホストが切替。単独で脱出した者だけが勝者になる勝利条件。
- **リザルトの“答え合わせ”演出**: ゲーム終了時に各ボットの性格を公開。
- **戦績保存**: 通算戦数・脱出数・勝利数・脱出率を localStorage に記録し、トップ画面に表示。

### 実装済み拡張（M3）
- **効果音**: Web Audio API でその場合成（音声ファイル不要）。行動・投票・死亡・脱出・嵐・チャット・勝敗に対応。🔊/🔇 でミュート切替（localStorage 保存）。
- **アニメ演出**: 日付スプラッシュ、資源トラックの数値ポップ、パネルのフェードイン、イベントバナー、ターゲット選択のパルス。
- **モバイル最適化**: レスポンシブ・グリッド、狭幅では「記録／チャット」をタブ切替、タップ領域の拡大。
- **サーバ側ランキング**: 名前単位の通算成績を JSON で永続化（`GET /leaderboard`）。トップ画面の「🏆 ランキング」から閲覧。
- **ルール説明**: トップ画面の「📖 遊び方」モーダルで初心者向けに解説。

### さらに後（フェイズ4以降）
- 木材集めのヘビ（病気）リスクの確率発生、天候デッキの多様化
- 同票時のリーダー裁定（現状は乱数）、フェイズ別タイマーUIの強化、絵文字リアクション
- 認証付きアカウント（ランキングの名前詐称対策）、AI強化（先読み／モンテカルロ）

## 4. アーキテクチャ

```
┌──────────────┐    Socket.IO (WebSocket)    ┌─────────────────────────┐
│  client      │ ◀──── redacted state ─────  │  server (authoritative) │
│ React + Vite │ ───── player intents ─────▶ │  Express + Socket.IO    │
└──────────────┘                             │  RoomManager / GameRunner│
                                             │   ├─ AI driver           │
        共有: @hellapagos/shared             │   └─ 権威 GameState      │
        （types / engine / content / ai）◀───┤  + 状態リダクション      │
                                             └─────────────────────────┘
```

- **サーバ権威モデル**: ゲーム状態の唯一の真実はサーバが持つ。クライアントは「意図（intent）」を送るだけ。チート・覗き見を構造的に防ぐ。
- **決定論的純粋エンジン**: `shared` の `engine` は `(state, action, rng) => state` の純粋関数群。サーバが適用し、AIも同じ関数で先読み評価できる。乱数はシードRNGを注入し再現可能。
- **状態リダクション**: ブロードキャスト前に、各クライアント向けに「他人の手札」を伏せた `PublicGameState` へ変換する。
- **モノレポ（npm workspaces）**: `shared` を `server`/`client` 双方から型ごと共有。`server` は `tsx`、`client` は Vite が TS ソースを直接解決するため、共有パッケージの別ビルドは不要。

### 技術選定理由
- **TypeScript 全面採用**: ルール（状態遷移）の正しさを型で守る。クライアント/サーバ/AIでロジックを1か所に集約。
- **Socket.IO**: ルーム機能・再接続・ブロードキャストが標準装備。議論や同時投票のリアルタイム性に最適。
- **React + Vite**: 状態駆動UI。HMRで開発が速い。

## 6. データモデル（要点）

```ts
type Phase = 'lobby' | 'weather' | 'action' | 'survival' | 'vote' | 'escape' | 'gameover'
type ActionType = 'fish' | 'water' | 'wood' | 'search'
type WeatherType = 'sunny' | 'rain' | 'storm'

interface Card { id: string; kind: 'food' | 'water'; amount: number }  // MVPは資源のみ

interface Player {
  id: string
  name: string
  isBot: boolean
  connected: boolean
  alive: boolean
  escaped: boolean
  sick: boolean              // 次ターン行動不能（フェイズ2のヘビ用）
  hand: Card[]               // 個人の隠し財産（探索で取得）
  pendingAction?: ActionType // その日に選んだ行動（解決待ち）
  contribute?: { food: number; water: number } // 生存判定で供出する量
  vote?: string | null       // 追放投票の対象 id（null=棄権）
  escapeVote?: boolean       // 脱出フェイズの賛否
}

interface GameState {
  phase: Phase
  day: number
  players: Player[]
  food: number; water: number; wood: number   // 共有トラック
  stormIn: number            // 嵐到来までの残り日数（0でその日が最終）
  weather: WeatherType | null
  firstPlayerIndex: number   // リーダー（同票裁定）
  shortage: number           // 生存判定での不足数（投票で減らす人数）
  rngState: number           // シードRNGの状態（決定論）
  log: LogEntry[]
  winners: string[]          // 脱出成功者 id
  config: GameConfig
}

interface GameConfig {
  raftWoodPerSeat: number    // 例: 2（木材2で1席）
  voyageProvisionPerSeat: number // 例: 1（席1につき水1・食料1を航海用に確保）
  initialStormIn: number     // 例: 7
  seed: number
}
```

### 公開状態（リダクション後）
他人の `hand` は枚数のみ公開し中身を隠す。`vote`/`pendingAction`/`contribute` は解決まで他人に伏せる（同時性の担保）。自分自身には完全な情報を返す。

## 7. ゲームエンジン（状態機械）

純粋関数群。各関数は新しい `GameState` を返す（イミュータブル更新）。

```
createGame(players, config) ─▶ lobby

startGame                   ─▶ weather
revealWeather               ─▶ action          // 天候適用、stormIn--
                                               // weather==storm かつ脱出条件未達なら gameover 予約

setAction(pid, action)                          // 全員 pendingAction が揃うまで action 維持
allActed? ─ resolveActions  ─▶ survival         // 行動を一括適用、トラック更新

computeSurvival             ─▶ 不足あり: vote / 不足なし: escape
  消費 = 生存者数。available = water/food + 任意供出。shortage = 必要 - 供給

setVote(pid, target) / setContribute(pid, ...)
allVoted? ─ resolveVotes    ─▶ 追放確定（最多票, 同票はリーダー裁定/乱数）→ escape

resolveEscape               ─▶ いかだ席 capacity = floor(wood / raftWoodPerSeat)
  capacity >= 生存者数 かつ 水・食料 >= 生存者数*voyageProvision のとき脱出可
  - 最終日(stormIn<=0): 乗れる人だけ脱出して勝利、他は敗北 → gameover
  - 通常日: 全員一致(または過半数)の脱出投票が可決なら脱出 → gameover、否決なら翌日へ
翌日へ ─▶ weather（day++）
```

**同時性**: 行動も投票も「全員が選び終えるまで結果を見せない」。サーバは全員の submit が揃うか、タイマー満了で締め切り、`resolve*` を呼ぶ。未提出のAI/人間にはデフォルト行動を補完。

## 8. 通信仕様（Socket.IO イベント）

### Client → Server
| イベント | ペイロード | 説明 |
|----------|-----------|------|
| `room:create` | `{ name }` | ルーム作成。`{roomId, playerId}` を返す |
| `room:join` | `{ roomId, name }` | 参加 |
| `room:addBot` | `{}` | AIボット追加（ホストのみ） |
| `room:removeBot` | `{ botId }` | ボット削除 |
| `game:start` | `{ config? }` | 開始（ホストのみ） |
| `action:choose` | `{ action }` | 行動選択 |
| `survival:contribute` | `{ food, water }` | 隠し財産の供出量 |
| `vote:cast` | `{ targetId \| null }` | 追放投票 |
| `escape:vote` | `{ leave: boolean }` | 脱出の賛否 |
| `chat:say` | `{ text }` | 議論チャット（フェイズ2） |

### Server → Client
| イベント | ペイロード | 説明 |
|----------|-----------|------|
| `room:state` | `RoomInfo` | ロビー情報（参加者一覧） |
| `game:state` | `PublicGameState` | リダクション済みの全体状態（毎更新） |
| `game:event` | `LogEntry` | アニメ/トースト用の単発イベント |
| `error` | `{ code, message }` | エラー通知 |

接続には `playerId` トークンを使い、リロード/再接続時に席へ復帰させる。

## 9. AI設計（MVP）

サーバの `GameRunner` が、AIプレイヤーの手番になったら `shared/ai` を呼ぶ。ヒューリスティック方針：

- **行動選択**: 共有トラックの欠乏度を評価し、最も逼迫した資源を補う行動を選ぶ。
  - 必要量 = 生存者数。`deficit(r) = need - stock`。`fish/water` は欠乏が大きい方、嵐が近く木材が足りなければ `wood`。手札が薄ければ確率的に `search`。
- **供出判断**: 自分が追放されるリスク（自分のせいで不足が出る度合い）が高いときだけ手札を供出。安全なら隠す（＝ニセ協力の再現）。
- **投票**: 「資源を供出しなかった者／何も貢献していない者」を優先的に追放対象にする単純スコアリング。乱数で多少ばらつかせる。
- **脱出投票**: 自分が乗れる（capacity内）なら賛成、乗れないなら反対しがち。

`ai.ts` は純粋関数 `aiChooseAction(view, persona, rng)` 等として実装し、人間UIと同じ intent を生成する。将来モンテカルロ等へ差し替え可能。

#### 性格付け（BotPersona — 実装済み）

ボットには4つの性格が割り当てられ（追加順に巡回）、供出・投票・アイテム使用が変化する。性格はサーバ側で秘匿（リダクションに含めない）。

| 性格 | 供出 | 投票で狙う相手 | アイテム |
|------|------|----------------|----------|
| 協力的 cooperative | 不足を素直に埋める | 供出が少ない者 | 解毒剤のみ |
| 溜め込み屋 hoarder | 半分しか出さない／探索を多用 | ライバルの溜め込み屋 | 解毒剤＋睡眠薬で自衛 |
| 狙撃手 sniper | 必要分のみ | 最も裕福な相手 | 拳銃で裕福な相手を射殺・略奪 |
| 臆病者 coward | 素直に埋める（疑われ回避） | 群れに同調（最少供出者） | 睡眠薬で自衛 |

加えて、投票フェイズではボットが性格に応じた**煽りセリフ**をチャットに流し、議論を盛り上げる（`aiChatLine`）。

## 10. 画面設計（UX）

1. **トップ/ロビー**
   - ルーム作成（名前入力）／URLで参加。参加者リスト、AIボットの追加・削除、開始ボタン（ホスト）。
2. **ゲーム盤**
   - 上部: 日数 / 嵐カウントダウン / 天候、共有トラック（食料・水・木材といかだ席ゲージ）。
   - 中央: 生存者カード一覧（生死・脱出・行動済み/投票済みのバッジ、自分の手札枚数）。
   - 下部: フェイズ別パネル。
     - 行動フェイズ: 4行動ボタン＋未確定者の人数表示。
     - 生存フェイズ: 不足数・供出スライダー。
     - 投票フェイズ: 各プレイヤーへ投票、締切タイマー。
     - 脱出フェイズ: 席数/必要数/賛否ボタン。
   - 右: ログ（時系列の出来事）。
3. **リザルト**: 脱出者・死亡者、勝敗、もう一度。

リアルタイム性が肝なので、各同時フェイズには**締切タイマー**を表示し、未提出はデフォルト処理で前進させる（オンラインで止まらない設計）。

## 11. 実装ロードマップ

- **M0（本MVP）**: モノレポ雛形 → 共有エンジン＋AI → サーバ（ルーム/進行/リダクション）→ クライアント（ロビー/盤/各パネル）。コアループがAI込みで一周する。
- **M1**: 特殊カード（拳銃・睡眠薬・解毒剤・ブードゥー・斧）、ヘビ病気、天候多様化、議論チャット＋タイマーUI。
- **M2**: 再接続復帰、観戦、ソロサバイバル・バリアント、演出（アニメ/SE）、戦績保存、モバイル最適化。
- **M3**: AI強化（先読み/性格付け）、ランクマッチ、リプレイ共有。

## 12. 非機能・運用メモ

- **テスト**: `shared/engine` は乱数を注入する純粋関数なので、シード固定でユニットテストしやすい（フェイズ遷移・生存計算・追放・脱出の境界）。
- **障害時の前進**: タイマー満了でデフォルト行動を補完し、1人が落ちてもゲームを止めない。
- **デプロイ**: サーバは Node プロセス（Fly.io/Render 等）、クライアントは静的ホスティング。WebSocket 維持のためスティッキー or 単一プロセス＋将来 Redis アダプタ。
